DOCUMENTO TÉCNICO — SALMA + BORRADO DEL MAPA
==============================================
Versión: 2.0 — 18 marzo 2026
Estado: referencia única para todo desarrollo


  ─────────────
  1. OBJETIVO
  ─────────────

Integrar SALMA como asistente inteligente dentro de Borrado del Mapa para:

- Descubrir lugares y recomendar experiencias
- Generar rutas con datos reales y verificables
- Sugerir alojamientos con precios y enlaces
- Editar itinerarios (manual + IA)
- Organizar viajes en el mapa del usuario
- Ofrecer info legal, cultural, histórica y práctica de cada destino
- Adaptarse a la nacionalidad y nivel de experiencia del viajero

Principio clave: SALMA piensa. La aplicación ejecuta.

La IA es capa de asistencia. La lógica principal reside en el sistema (frontend + backend).

  ──────────────────────────
  3. ARQUITECTURA GENERAL
  ──────────────────────────

El sistema se divide en seis componentes:

1. SALMA conversacional (salma-prompt — 9 bloques)
2. Editor visual de rutas (UI + lógica de aplicación, sin IA)
3. SALMA ayudante del editor (Prompt Editor — diseñado, no implementado)
4. Orquestador de intención (Prompt Orquestador — diseñado, no implementado)
5. Base de conocimiento KV (3 niveles)
6. Detección de destino y categorías (en el Worker)

Flujo actual (V1):

  Usuario
    ↓
  Worker recibe mensaje
    ↓
  Detecta país + categorías (keywords)
    ↓
  Lee KV: ficha país (N1) + destinos (N2) si hay datos
    ↓
  Ensambla prompt: salma-prompt + contexto dinámico
    ↓
  Llama a Claude API
    ↓
  Extrae ruta (SALMA_ROUTE_JSON) si la hay
    ↓
  Devuelve { reply, route } al frontend

Flujo futuro (con orquestador):

  Usuario
    ↓
  Orquestador de intención
    ↓
  Decide entre:
    → SALMA conversacional
    → Editor visual (sin IA)
    → SALMA ayudante del editor
  ↓
  Base de datos de rutas (Firestore)


  ──────────────────────────────────────────
  4. BASE DE CONOCIMIENTO — CLOUDFLARE KV
  ──────────────────────────────────────────

Binding en Worker: SALMA_KV
Si SALMA_KV no está bindeado, el Worker funciona sin contexto de destino.

4.1 NIVEL 1 — Ficha país (estático, 193 países)

  Clave KV:     dest:{código}:base
  Ejemplo:      dest:vn:base

  Contenido por país:
  - Capital, idioma oficial, idioma útil para viajero
  - Moneda, cambio aprox EUR, huso horario, prefijo tel, enchufes
  - Visados para 10-15 nacionalidades principales (ES, FR, DE, UK, US, MX, AR, CO, BR, AU...)
  - Mejor época / época a evitar
  - Seguridad (nivel 1-5 + comentario)
  - Vacunas, agua potable, emergencias
  - Coste diario mochilero / medio (EUR)
  - Propinas, curiosidad viajera
  - Historia del país (resumen práctico para viajero)
  - Cultura (costumbres, tabúes, vestimenta, religión)
  - Legal viajero:
    · Leyes que pillan a turistas (drogas, alcohol, vestimenta, foto, drones, VPN)
    · Qué pasa si te detienen (derechos, contacto consular)
    · Aduanas (efectivo, tabaco, alcohol, medicamentos)
    · Conducción (carnet internacional, seguro, multas)
    · Estafas legales comunes (timos policiales, sobornos)
    · Seguros de viaje obligatorios
    · Derechos del viajero en aeropuertos
    · Leyes LGBTQ+
    · Nómada digital (legal o no, visados específicos)
    · Disclaimer: "info orientativa, consulta tu embajada"
  - Keywords de búsqueda (ciudades, variantes de nombre)

  Generador:    generate.js
  Coste:        ~$0.90 (193 llamadas × ~1500 tokens)
  Subida:       upload-kv.js

  Keywords indexadas:
  Clave KV:     kw:{keyword-normalizado} → código de país
  Ejemplo:      kw:hanoi → vn

4.2 NIVEL 2 — 10 destinos por país (estático, 193 países)

  Clave KV:     dest:{código}:destinos       (todos los destinos del país)
                dest:{código}:spot:{slug}    (destino individual)
  Ejemplo:      dest:vn:destinos
                dest:vn:spot:hanoi

  Contenido por destino:
  - id (slug), nombre, tipo, región
  - Descripción (2-3 frases, sin paja)
  - Días recomendados
  - Mejor época
  - Cómo llegar (medio, duración, coste EUR)
  - Dónde dormir: mochilero / medio / comfort (tipo + rango EUR/noche)
  - Qué hacer (actividades con precio)
  - Dónde comer (platos, precios, zonas)
  - Consejo local (tip de alguien que ha estado)
  - Plan B lluvia
  - Keywords de búsqueda

  Índice de slugs:
  Clave KV:     spot:{slug} → {código}:{slug}
  Ejemplo:      spot:hanoi → vn:hanoi

  Generador:    generate-nivel2.js
  Coste:        ~$11.50 (193 llamadas × ~4000 tokens)
  Subida:       upload-kv-nivel2.js

  Total: ~1.930 destinos, ~7.000 keywords, ~3.3 MB en KV

4.3 NIVEL 3 — Contexto en vivo (dinámico)

  No es KV estático. Se activa cuando Salma detecta que necesita dato fresco.

  Incluye:
  - Noticias locales y alertas de seguridad
  - Precios reales de vuelos (Kiwi en F4) y hoteles (Trivago en F4)
  - Eventos y festivales con fechas del año actual
  - Info hiperlocal (SIMs, fronteras, estado de carreteras)
  - Regulaciones recientes (visados, requisitos sanitarios, tasas)

  Detección inteligente:
  - Llamada clasificadora rápida (modelo barato) que decide si buscar o no
  - Si dice "necesito buscar: [query]" → Worker hace búsqueda web
  - Si dice "no necesito buscar" → responde con N1 + N2

  Caché de búsquedas:
  Clave KV:     cache:{query-hash} → resultado + timestamp
  TTL:          7 días
  Si otro usuario pregunta lo mismo → respuesta directa sin búsqueda

  Feedback de usuarios:
  Clave KV:     feedback:{código}:{destino}:{tema} → corrección + timestamp
  Si se acumulan 2-3 feedbacks iguales → ficha marcada "pendiente actualizar"

  Contador de demanda:
  Clave KV:     demand:{código} → número de consultas
  Prioriza qué países ampliar de N1 a N2

  Estado: diseñado, pendiente de implementar

  ──────────────────────────────────────
  6. SALMA-PROMPT (PRODUCCIÓN — V1)
  ──────────────────────────────────────

Archivo: salma-prompt-v1.txt
Worker:  salma-worker-v1-final.js

9 bloques ensamblados en el system prompt:

BLOQUE 1 — IDENTIDAD
  Andaluza cercana, tutea, multiidioma manteniendo carácter.

BLOQUE 2 — PERSONALIDAD Y TONO
  Cada frase lleva dato útil. Opinión propia, se moja.
  Gustos: Extremoduro, Springsteen, Sabina, AC/DC. Anti-reguetón.
  Recursos narrativos del espíritu de las letras (no citas textuales).
  No acepta machismo ni sexismo.

BLOQUE 3 — MULETILLAS (USO MEDIDO)
  1 cada 8-10 mensajes, solo cuando encaja emocionalmente.
  5 categorías: planear, improvisar, social, problemas, cerrar decisiones.
  Nunca forzadas, nunca repetidas seguidas.

BLOQUE 4 — REGLAS ANTI-PAJA
  Frases prohibidas listadas. Adjetivos vacíos prohibidos.
  3 pares de ejemplo bueno vs malo (vacunas, Khao San Road, Vietnam moto).
  Test: sin personalidad, ¿queda info útil? Si no → paja.

BLOQUE 5 — TONO SEGÚN CONTEXTO Y NIVEL USUARIO
  Graciosa para planificar, seria para seguridad/legal/LGBTQ+.
  Detecta nivel: novato (más contexto) vs curtido (al grano).
  Usa historial para no contradecir gustos/presupuesto del usuario.

BLOQUE 6 — INFORMACIÓN Y DATOS
  Datos veraces y contrastables. Nunca inventar.
  Cubre: visados por nacionalidad, salud, transporte, comida,
  alojamiento, historia, cultura, legal viajero.
  Disclaimer en temas legales.

BLOQUE 7 — FORMATO DE RESPUESTA
  Con ruta: 1-2 frases, detalle en SALMA_ROUTE_JSON.
  Conversacional: extenso pero denso. Prosa fluida.
  Nunca: bullets, markdown ###, coordenadas en chat, emojis excesivos.

BLOQUE 8 — MODOS Y FORMATO SALMA_ROUTE_JSON
  3 modos: Explorador / Descubridor / Ruta.
  Puntos verificables (Google Maps, Booking).
  Nombres exactos para enlaces a Google Maps.
  Datos para mapa: tipo, nombre, zona, coords, duración, url_reserva.
  Rutas organizadas por días.
  JSON con stops: name, headline, description, narrative, type, day (int), lat, lng.
  day SIEMPRE número entero, nunca string.

BLOQUE 8B — MAPA, TARJETAS, ALOJAMIENTO Y NAVEGACIÓN
  Mapa personal del usuario (lugares, restaurantes, experiencias, miradores, hoteles, rutas).
  Tarjetas visuales: lugar (nombre, desc, duración, ver ruta/guardar) y hotel (nombre, zona, desc, reservar/guardar).
  Alojamiento: pregunta fechas/presupuesto/zona, varias opciones, afiliados Booking.
  Funciones inteligentes: cercanos a guardados, optimizar rutas, sugerir zonas dormir.
  Sugerencias automáticas: hasta 3 lugares cerca de su ruta.
  Mapa por día: cada día visualizable independiente.
  Edición: añadir/quitar/reordenar desde móvil y web.
  Navegación externa: Google Maps / Apple Maps.

  ─────────────────────────────────
  9. ESTRUCTURA DE DATOS — RUTAS
  ─────────────────────────────────

Todas las rutas se guardan en Firestore con estructura JSON.

  {
    "id": "ruta_123",
    "titulo": "Granada en 3 días",
    "destino": "Granada, España",
    "dias": [
      {
        "dia": 1,
        "zona": "Centro histórico",
        "paradas": [
          {
            "id": "p1",
            "tipo": "lugar",
            "nombre": "Catedral de Granada",
            "descripcion": "Visita cultural imprescindible",
            "duracion_min": 90,
            "lat": 37.176,
            "lng": -3.599
          }
        ]
      }
    ],
    "hotel_base": null,
    "notas": "",
    "updated_at": ""
  }

Este JSON se usa para:
- Renderizar rutas en el editor
- Pintar el mapa (Leaflet)
- Editar paradas (drag & drop)
- Enviar contexto a SALMA (current_route en el Worker)
- Guardar cambios en Firestore

  10. FLUJOS DEL SISTEMA
  ──────────────────────────────

10.1 Crear ruta (ACTIVO)

  Usuario envía mensaje → Worker → Detecta país/categoría →
  Lee KV (N1+N2 si hay) → Ensambla salma-prompt + contexto →
  Claude genera ruta → Worker extrae SALMA_ROUTE_JSON →
  Frontend renderiza mapa + tarjetas → Usuario guarda en Firestore

10.2 Conversación sin ruta (ACTIVO)

  Usuario pregunta (vacunas, visados, cultura...) → Worker →
  Detecta país/categoría → Lee KV → Salma responde en prosa
  (sin SALMA_ROUTE_JSON) → Frontend muestra texto

10.3 Edición manual (ACTIVO)

  Usuario abre ruta → Editor visual → Drag & drop / añadir / eliminar →
  Actualizar JSON → Guardar en Firestore

10.4 Edición con IA (DISEÑADO, NO ACTIVO)

  Usuario pulsa "Optimizar día" → Enviar ruta al Prompt Editor →
  SALMA devuelve acciones JSON → Mostrar preview →
  Usuario acepta → Actualizar JSON → Guardar

10.5 Búsqueda web en vivo — Nivel 3 (DISEÑADO, NO ACTIVO)

  Usuario pregunta algo cambiante → Worker → Llamada clasificadora
  (modelo barato) → "necesito buscar: [query]" → Worker busca web →
  Cachea resultado en KV (7 días) → Inyecta en contexto →
  Claude responde con dato fresco
